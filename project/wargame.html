02. My 5 Key Challenges

제가 직접 설계하고 출제한 5가지 핵심 보안 문제의 워크스루입니다. (화살표를 클릭하여 단계를 확인하세요)

01. Cookie Manipulation Authentication Bypass

클라이언트 측 쿠키 값을 서버에서 재검증하지 않고 그대로 신뢰하는 쿠키 변조 취약점입니다. 서버가 로그인 여부를 오직 loggedin=1이라는 쿠키 값만으로 판단하기 때문에, 공격자가 개발자 도구에서 이 값을 조작하여 손쉽게 관리자 권한을 탈취할 수 있는 구조를 설계했습니다.

1 / 4
[Stage 1] 문제 화면
"You are not logged in" 힌트를 통해 인증 세션 혹은 쿠키와 관련된 취약점임을 유추할 수 있습니다.
2 / 4
[Stage 2] 개발자 도구(F12) 분석
인증 관련 쿠키인 loggedin의 Value 값이 0(거부)으로 설정되어 있는 것을 확인했습니다.
3 / 4
[Stage 3] 데이터 변조 및 적용
값을 1(관리자 권한)로 직접 변경 후 페이지를 새로고침하여 서버 인증을 통과시켰습니다.
4 / 4
[Stage 4] Flag 획득
인증 우회 성공 후 최종 플래그 FLAG{first_cookie_1234@@}를 획득했습니다.
❮ ❯

02. Path Traversal LFI Vulnerability

서버에서 외부 입력값을 검증하지 않고 파일 경로로 사용할 때 발생하는 LFI(Local File Inclusion) 취약점입니다. 상위 디렉토리로 이동하는 ../(Dot-Dot-Slash) 패턴을 활용해 서버 내부의 민감한 설정 파일이나 플래그 파일에 접근하는 공격 리스크를 설계했습니다.

1 / 3
[Stage 1] 문제 접속 및 탐색
2 / 3
[Stage 2] 소스 코드 분석 (View Source)
페이지 소스 보기 중 주석으로 숨겨진 hint: Next flag is in /etc/star_webpass/cmd를 발견하여 타겟 경로를 식별했습니다.
3 / 3
[Stage 3] 경로 조작 공격 및 Flag 획득
단순히 URL에 ?page=/etc/star_webpass/cmd 를 입력시 flag 가 있는 페이지가 나오지 않지만, ../../를 반복 사용하여 URL에 넣다보면 FLAG{LFI_master_00002shaha}를 획득합니다.
❮ ❯

03. PHP strcmp() Vulnerability Type Juggling

PHP의 strcmp() 함수가 문자열이 아닌 배열([ ]) 형태의 입력을 받았을 때 발생하는 취약점입니다. 함수가 배열을 처리하지 못해 NULL(0)을 반환하는 특성을 이용하여, 정확한 패스워드를 몰라도 인증 로직을 무력화하고 권한을 획득하는 시나리오를 설계했습니다.

1 / 5
[Stage 1] 문제 확인
패스워드 입력값이 서버 측의 strcmp() 함수를 통해 검증되는 단계입니다.
2 / 5
[Stage 2] 일반적인 접근 시도
임의의 패스워드 입력 시 "Wrong password, try again" 알림창이 뜨며 인증에 실패합니다.
3 / 5
[Stage 3] 프록시 인터셉트 (Burp Suite)
Burp Suite를 통해 전송되는 패킷을 캡처합니다. 현재 password=asdf와 같은 문자열 형태로 전송되고 있습니다.
4 / 5
[Stage 4] 변수 타입 변조 (Array Injection)
파라미터명을 password[]로 수정하여 배열 타입으로 전송합니다. 이를 통해 함수가 0(True)을 반환하도록 유도합니다.
5 / 5
[Stage 5] 인증 우회 및 Flag 획득
검증 로직이 무력화되면서 최종 플래그 FLAG{strcmp_bypass_success_9988@@}를 획득했습니다.
❮ ❯

04. Encoding Logic Exposure Information Disclosure

서버의 암호화 로직이 소스 코드에 노출되어 발생하는 인코딩 방식 유출 취약점입니다. 단방향 해시가 아닌 Base64 -> strrev -> bin2hex 순서의 가역적인 인코딩 조합을 사용함으로써, 공격자가 로직을 분석하여 원래의 Secret 값을 역연산으로 도출해낼 수 있는 위험성을 설계했습니다.

1 / 6
[Stage 1] 문제 확인 및 힌트 수집
문제 화면에서 encodeSecret() 함수 로직을 이용하라는 핵심 힌트를 확인했습니다.
2 / 6
[Stage 2] 소스 코드 분석 (View Source)
노출된 코드에서 인코딩된 문자열과 처리 순서를 파악했습니다.
Logic: bin2hex(strrev(base64_encode($secret)))
3 / 6
[Stage 3] 역연산 Step 1: Hex Decoding
가장 마지막 단계인 Hex를 먼저 복구합니다.
hex2bin("3d4d31554242315242786b52") -> =M1UBB1RBxkR
4 / 6
[Stage 4] 역연산 Step 2: String Reverse
중간 단계인 문자열 뒤집기를 수행합니다.
strrev("=M1UBB1RBxkR") -> RkxBR1BBU1M
5 / 6
[Stage 5] 역연산 Step 3: Base64 Decoding
첫 인코딩 단계인 Base64를 복구하여 원본 문자열을 획득합니다.
base64_decode("RkxBR1BBU1M") -> FLAGPASS
6 / 6
[Stage 6] 인증 성공 및 Flag 획득
도출해낸 FLAGPASS를 입력하여 최종 플래그
FLAG{encodeSecret_function_bypass_success_2025}를 획득했습니다.
❮ ❯

05. Heartbleed Simulation Memory Over-read

OpenSSL의 유명 취약점인 Heartbleed를 재현한 메모리 오버리드(Memory Over-read) 취약점입니다. 사용자가 전송한 데이터의 실제 길이와 관계없이 사용자가 지정한 length 값만큼 메모리를 반환하는 서버 측 로직의 허점을 이용하며, 이를 통해 서버 메모리에 남아있는 민감한 데이터(Flag)를 유출시키는 시나리오를 설계했습니다.

1 / 4
[Stage 1] 문제 확인 및 힌트 분석
"서버에 너무 많은 것을 요구하면 심장이 비밀을 드러낸다"는 힌트를 통해, 데이터 길이를 속여 메모리를 유출시키는 Heartbleed 공격 방식임을 유추했습니다.
2 / 4
[Stage 2] 1차 메모리 유출 시도 (Probe)
Payload를 a(1바이트)로 입력하고 Length를 840으로 높게 설정하여 요청했습니다. 출력된 메모리 끝부분에서 FLAG 문구의 시작점을 포착했습니다.
3 / 4
[Stage 3] 2차 메모리 유출 및 데이터 식별
더 넓은 범위의 데이터를 얻기 위해 Length를 890으로 증액하여 재요청했습니다. 그 결과, 서버 메모리 깊숙한 곳에 저장된 전체 플래그 값을 추출하는 데 성공했습니다.
4 / 4
[Stage 4] Flag 제출 및 성공
추출한 FLAG{thirdstar_heartbleed_memory_leak_success_2025}를 제출하여 문제 해결을 완료했습니다.
❮ ❯
